HttpSession RUBiS based on Vanilla RUBiS EJB_local_remote_CMP2.0

 

 

1. Objectives

l      Vanilla RUBiS has the implicit login mechanism. Every time user putting a bid, putting a comment to the other user, buying an item, selling an item and looking oneÅfs own information, he or she must log in the system. This behavior is not common and not user-friendly.
Usually web application has the explicit login mechanism. At the beginning of a session user explicitly logs in the system. It keeps login state until he or she logs out explicitly. After login, putting a bid, putting a comment, etc. never require to login again.
In order to realize such a common functionality in RUBiS we replace the implicit login mechanism with the explicit login mechanism.

l      In vanilla RUBiS all EJBs are archived in a jar file called rubis.jar. Jar file is an unit of undeploying and deploying an application component. In order to realize micro-reboot of each EJB we should archive each EJB separately.

 

2. Explicit Login Mechanism

 

2.1 Login Page and URL Transitions

 

Figure 1 illustrates transition of URLs in Vanilla RUBiS EJB_local_remote_CMP2.0. The servlets PutBidAuth, PutCommentAut and BuyNowAuth and html files sell.html and about_me.html indicate a login page, which always requires inputting the registered nickname and password. Input nickname and password are verified at the next servlets PutBid, PutComment, BuyNow, BrowseCategories and AboutMe respectively.

Figure 2 illustrates transition of URLs in modified RUBiS, call it as HttpSession RUBiS. HttpSession RUBiS has an explicit login page, which requires inputting nickname and password. The next servlet Login verifies them. A user request for putting a bid comes to PutBid servlet instead of PutBidAuth. If the user has already logged in using the explicit login page, PutBid servlet proceeds putting the bid. If the user hasnÅft logged in yet, PutBid redirects its request to the explicit login page. After verifying nickname and password, the request returns to PutBid servlet and PutBid proceeds putting the bid. The other four implicit login mechanisms are replaced similarly in HttpSession RUBiS.

 

2.2 Session State Management

 

In vanilla RUBiS PutBid servlet verify input nickname and password accessing the user account database. If they are correct, PutBid servlet returns his or her user id with a bidding form to his or her web browser. His or her web browser sends the bidding information with his or her user id as an URL parameter to the next servlet StoreBid. The user id passed to the client indicates state of this session. Vanilla RUBiS always passes each sessionÅfs current state to its client. This technique is widely used when a life time of the session state is short, like between two or three servlets. But in HttpSession RUBiS the life time of user id is relatively long. Therefore we use the http session object as a vehicle of maintaining each session state. 

 

2.2.1 Http Session Object in HttpSession RUBiS

 

Whenever HttpSession RUBiS receives a request to one of its pages, it checks existence of the http session object associated with its user. If there is no http session object, HttpSession RUBiS creates a http session object and returns its associated cookie named JSESSIONID to the client. After receiving a cookie from the server, the client attaches the cookie to requests and sends them to the server whenever the cookie hasnÅft been expired yet. When a request equipped with a cookie JSESSIONID comes to the server, it find the session object associated with the cookie.

 

We keep the following four data in the http session object as the attributes.

 

Attribute name

Description

USERID

Id of the currently accessing user

ITEMID

Id of the item targeted for putting a bid, buying, selling, viewing

TO

Id of the user targeted for putting a comment

URL

Return URL after logging in

 

2.2.2 Login and user id

 

Login.html in Figure 2 indicates login form. Its successor servlet Login receives nickname and password from the client and verifies them accessing user account database. If they are correct, the corresponding user id is extracted from database and it is set to the attribute USERID. After that Login servlet extracts the value of the attribute URL. If it is null, Login servlet indicates welcome message to the client. If it is not null, Login servlet redirects the request to the URL specified by extracted value from the attribute URL. The attribute URL is set at PutBid, PutComment, BuyNow, BrowseCategories or AboutMe. See also 2.2.5.

 

2.2.3 Logout

 

Logout servlet remove the http session object associated with the user. Therefore all of the four attributes disappear. The server also sets the cookie of the user expired and sends it to the client.

 

2.2.4 Navigation Header

 

USERID is referred by all servlets. Each servlet indicate the page navigation header. This header consists of six items. The first item is a link to login or logout. It depends on the value of the attribute USERID. Each servlet extracts a value of the attribute USERID from its http session object. If the value is the positive integer, it indicates the link to Logout servlet. If the value is null, it indicates the link to login.html.

 

2.2.5 PutBid, PutComment, BuyNow, BrowseCategories and AboutMe

 

PutBid servlet extracts a value of the attribute USERID. If its value is null, PutBid sets the item id which came from the client as the parameter of URL and the servlet relative URL Ågservlet/edu.rice.rubis.beans.servlets.PutBidÅh to the attribute ITEMID and URL respectively and redirects the request to login.html. After the authentication in Login servlet the request comes back to PutBid servlet again. Then PutBid extracts the value of the attribute USERID and ITEMID and indicates the bidding form for the item to the client. If the extracted value from the attribute ITEMID is not valid, PutBid indicates an error message to the client.

The behaviors of the servlets PutComment, BuyNow, BrowseCategories and AboutMe are similar to the above. PutComment servlet set the selected user id, which identifies the target user for putting a comment, to the attribute TO in addition to the item id and the return URL before going to login.html. BrowseCategories and AboutMe only set the return URL to the attribute URL instead of the item id and the return URL before going to login.html.

Note that BrowseCategories servlet has two different behaviors. Both indicate all of the categories. But one is for just browsing them and the other is for selling an item. The latter case must be supplied two parameters nickname and password as its parameters. And only the latter case checks the value of the attribute USERID and so on as described above.
